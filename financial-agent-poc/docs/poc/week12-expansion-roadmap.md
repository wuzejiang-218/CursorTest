# 第 12 周 PoC 验收后扩容路线图（草案）

## 1. 验收门槛回顾

以 [`01-scope-and-acceptance.md`](01-scope-and-acceptance.md) 为唯一验收依据；未达标项进入「试点准入条件」而非直接扩容。

## 2. 试点阶段（建议再 3 个月）

### 2.1 能力扩容

- 将 `ApiRunner` 替换为真实 HTTP 客户端 + 契约校验（Schemathesis / Dredd 等择一）。
- `UiRunner` 接入 Playwright，并与测试数据工厂绑定。
- `DataValidator` 接入只读从库 + 断言模板库。

### 2.2 智能体层

- Parser/CaseGen 引入 LLM + **强 Schema** 输出与自动校验失败重试。
- 引入「人工复核门禁」：高风险步骤强制 HITL。

### 2.3 数据与平台

- 结果持久化到 PostgreSQL；报告中心与现有测试门户集成。
- OpenTelemetry 全链路追踪；与 ELK/ClickHouse 对接。

## 3. 组织与流程

- 建立规则版本与测试基线的联合发布流程（风控规则变更必带回归包）。
- 质量度量纳入部门级看板（成本、耗时、漏检、误报）。

## 4. 企业级前置（6–12 个月视角）

- 多租户隔离、密钥管理（KMS）、模型供应商 SLA 与离线降级策略。
- 红队与提示注入防护基线；供应链（依赖）扫描纳入 CI。

## 5. Go / No-Go 决策表（简版）

| 维度 | 试点 Go 条件 |
|------|----------------|
| 指标 | 连续 4 周达到或优于 PoC 验收阈值 |
| 稳定性 | 流水线非环境性失败率 < 5% |
| 合规 | 审计与脱敏抽查零严重问题 |
| 成本 | 单版本发布周期内 token/调用成本在预算内 |
