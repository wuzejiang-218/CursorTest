## 三、关键风险提示

### 1) 字段映射是最大重灾区（15 条 / 25%）

涉及 **3 人共同维护**（肖 7、蒋 5、胡 3），跨多个产品类型（ELN / Sharkfin / Phoenix / Step-down 等），建议：

- 沉淀**前端 → 后端 → 订单详情** 三段式字段映射表，作为提测前 checklist。
- 各产品类型单独维护字段单元测试，覆盖 KO Type、Format、Direction 等容易遗漏的枚举。

### 2) 邮件通知由蒋杰独立承担（8 条）

模板缺字段（8069 / 8089 / 8091 / 8093）属于发版前可拦截的低级问题，建议：

- 邮件模板沉淀字段 checklist：TaponID / Issuer / size 单位 / 千分符 / logo / 机构链接 / 布局规范。
- 通知逻辑独立做团队过滤 / 去重单测（针对 8063 / 8064 这类边界问题）。

### 3) 订单 ↔ Tapon 联动是肖赣丰独立战场（5 条）

主单修改影响范围控制不严，建议：

- 主单修改条款 / ISIN / Valoren / FixingPrice / TS 五种场景做端到端用例。
- 批量审批并发场景（#8127）需要单独压测验证。

### 4) 上架失败 4 条覆盖 ELN / Sharkfin / Phoenix（胡明明）

建议增加**产品级冒烟用例**，每发布前对所有支持的产品类型跑一次上架。

### 5) 拼单状态机口径需统一（肖 + 彭超 8 条）

- 募集金额统计口径（pending / accepted / invalid / expired）需统一文档化。
- lastlook / 报价撤销 / 主单修改三类触发下架的条件需要矩阵化梳理（避免出现 #8022 全部下架、#8073 不应下架这种"误触发 / 漏触发"对立 Bug）。

### 6) 重复 Bug 链需关注根因

- #8023 → #8020：均涉及"已撤销报价对应 Tapon 卡片下架"
- #8052 → #8036：均涉及"募集金额统计未剔除无效状态"

两条都由彭超关联，建议确认是否同一处代码导致，避免后续重复提单。
